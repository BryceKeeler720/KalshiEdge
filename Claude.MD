## Key Technical Decisions

### Kalshi API
- **Single unified API** â€” one base URL handles market data, trading, and portfolio
- **Base URLs**: Production `https://trading-api.kalshi.com/trade-api/v2`, Demo `https://demo-api.kalshi.co/trade-api/v2`
- **Auth**: RSA-PSS signed requests. Generate API key + download private key PEM from Kalshi dashboard. Each request is signed with `timestamp + method + path` using the RSA private key.
- **Prices are in CENTS as integers** â€” a price of 65 means $0.65 (65% probability). Range: 1-99.
- **Settlement**: USD, not crypto. Winners get $1.00 per contract ($100 in cents). Losers get $0.
- **Demo environment**: Full sandbox at `demo-api.kalshi.co` with fake money and real-ish market data. Always develop here first.

### Authentication Pattern
```python
from kalshi_python import Configuration, KalshiClient

config = Configuration(
    host="https://demo-api.kalshi.co/trade-api/v2"  # or production URL
)

with open(KALSHI_PRIVATE_KEY_PATH, "r") as f:
    private_key = f.read()

config.api_key_id = KALSHI_API_KEY_ID
config.private_key_pem = private_key

client = KalshiClient(config)
```

If using `kalshi-python` SDK has issues, fall back to raw `httpx` with manual RSA-PSS signing:
```python
import base64, time
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.hazmat.primitives.asymmetric import padding

def sign_request(private_key_pem: str, timestamp: str, method: str, path: str) -> str:
    key = serialization.load_pem_private_key(private_key_pem.encode(), password=None)
    message = f"{timestamp}{method}{path}".encode()
    signature = key.sign(message, padding.PSS(
        mgf=padding.MGF1(hashes.SHA256()),
        salt_length=padding.PSS.MAX_LENGTH
    ), hashes.SHA256())
    return base64.b64encode(signature).decode()
```

### Order Execution Rules
- **Always use limit orders** â€” resting limit orders are fee-free on Kalshi. Taker (market) orders incur fees.
- Price in **integer cents** (1-99). `yes_price=55` means buy YES at 55Â˘.
- `action`: `"buy"` or `"sell"`
- `side`: `"yes"` or `"no"`
- `type`: `"limit"` (preferred) or `"market"`
- `count`: number of contracts (integer)
- `expiration_ts`: optional Unix timestamp for GTD orders (omit for GTC)
- **No heartbeat required** â€” unlike Polymarket, Kalshi orders persist until filled or cancelled
- Cancel with `DELETE /portfolio/orders/{order_id}`

### Forecaster Design
- Use `anthropic.Anthropic()` wrapped in `wrap_anthropic()` from 2signal
- Model: `claude-sonnet-4-6` for forecasting, `claude-haiku-4-5-20251001` for news screening
- Run 3 forecasts per market at temperatures 0.3, 0.5, 0.7
- Aggregate via trimmed mean, apply extremization (factor 1.3)
- **Convert Kalshi cents to probability before comparing**: `p_market = yes_price / 100`

## Common Pitfalls

- **Prices are CENTS (integers), not decimals** â€” 65 means $0.65. Convert to probability: `p = price / 100`. When placing orders, `yes_price` takes int 1-99.
- **Tickers, not token IDs** â€” Kalshi uses human-readable tickers like `FED-RATE-CUT-JUN26`, not opaque token IDs.
- **Events contain markets** â€” an event like "Fed rate decision" has multiple markets for different outcomes. Use `GET /events` to understand groupings.
- **Auth tokens expire every 30 minutes** â€” handle refresh in the client wrapper.
- **Rate limits return 429** â€” implement exponential backoff. Kalshi uses tiered rate limits.
- **Demo vs Prod are separate** â€” different API keys, different base URLs, separate account balances. Always test in demo first.
- **No WebSocket auth in demo** â€” WebSocket works differently in demo vs production.
- **Balance is in cents** â€” `balance.balance` returns cents. Divide by 100 for dollars.
- **Extremization is critical** â€” raw LLM probabilities hedge toward 50%.
